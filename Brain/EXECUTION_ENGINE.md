# EXECUTION_ENGINE.md

## The Execution Engine: From Plan to Action

The Execution Engine is the operational core of BugHunterAI, responsible for translating strategic plans generated by the Planning Engine into concrete, verifiable actions within the target environment. It acts as the bridge between abstract objectives and tangible results, ensuring that every task is performed efficiently, robustly, and with meticulous logging.

### Purpose

The primary purpose of the Execution Engine is to:

1.  **Orchestrate Task Execution**: Systematically execute individual tasks and subtasks defined by the Planning Engine.
2.  **Manage Resources**: Allocate and manage computational resources (e.g., CPU, memory, network bandwidth) for various tools and processes.
3.  **Handle Failures**: Implement robust error handling, retry mechanisms, and fallback strategies to ensure resilience.
4.  **Capture Outputs**: Collect, parse, and standardize the outputs from executed tools and actions.
5.  **Provide Observability**: Log all execution steps, inputs, outputs, and status changes for auditing and real-time monitoring.
6.  **Ensure Idempotency**: Design tasks to be repeatable without causing unintended side effects.

### Architecture

The Execution Engine is designed with modularity and scalability in mind, comprising several key components:

-   **Task Dispatcher**: Receives prioritized tasks from the Planning Engine and assigns them to appropriate executors.
-   **Tool Adapters**: Standardized interfaces for interacting with various external security tools (e.g., Subfinder, Nuclei, httpx). Each adapter translates generic task commands into tool-specific invocations and parses their raw outputs into a structured format.
-   **Browser Automation Module**: Integrates with the Browser Architecture to perform web-based interactions, data extraction, and vulnerability testing (e.g., DOM XSS, business logic flaws).
-   **Terminal Automation Module**: Manages command-line tool execution, including multi-terminal execution for parallel processing.
-   **Resource Manager**: Monitors system load and available resources, preventing over-utilization and ensuring stable operation.
-   **Logging and Telemetry**: A centralized system for recording all execution events, performance metrics, and error details.
-   **State Manager**: Persists the state of ongoing tasks, allowing for graceful recovery from interruptions and tracking progress.

### Execution Flow

1.  **Task Ingestion**: The Execution Engine receives a `Task` object from the Planning Engine, which includes:
    -   `task_id`: Unique identifier.
    -   `tool_name`: The tool or module to be used (e.g., `subfinder`, `browser_scraper`).
    -   `parameters`: Key-value pairs for tool configuration.
    -   `expected_output_schema`: Defines the structure of the expected output.
    -   `retry_policy`: Rules for retrying failed tasks.
    -   `timeout`: Maximum execution time.

2.  **Pre-execution Validation**: Before execution, the engine performs checks:
    -   **Tool Availability**: Verifies if the specified tool is installed and configured.
    -   **Input Validation**: Ensures parameters conform to the tool's requirements.
    -   **Resource Check**: Confirms sufficient resources are available.

3.  **Tool Invocation**: The Task Dispatcher selects the appropriate Tool Adapter or Automation Module and invokes the tool with the provided parameters. This often involves generating shell commands, API calls, or browser scripts.

4.  **Output Capture and Parsing**: Raw output from the tool is captured and transformed into a structured `Observation` object, which includes:
    -   `observation_id`: Unique identifier.
    -   `task_id`: Reference to the originating task.
    -   `status`: `SUCCESS`, `FAILURE`, `TIMEOUT`.
    -   `raw_output`: Original tool output.
    -   `parsed_data`: Structured data extracted from the output.
    -   `error_details`: If applicable.
    -   `timestamp`.

5.  **Post-execution Analysis**: The `Observation` is then sent back to the Reasoning Engine and Memory System for further analysis, evidence correlation, and potential replanning.

### Error Handling and Resilience

The Execution Engine is designed to be fault-tolerant:

-   **Retries with Backoff**: Failed tasks are automatically retried with increasing delays.
-   **Fallback Strategies**: If a primary tool fails repeatedly, alternative tools or methods are attempted (e.g., if `subfinder` fails, try `amass`).
-   **Circuit Breakers**: Tools or services exhibiting high failure rates are temporarily isolated to prevent cascading failures.
-   **Dead-Letter Queue**: Unrecoverable tasks are moved to a dead-letter queue for manual inspection.

### Integration with Other Components

-   **Planning Engine**: Receives tasks and sends back execution status and observations.
-   **Memory System**: Stores execution logs, tool outputs, and state information.
-   **Knowledge System**: Consults tool documentation and best practices for optimal invocation.
-   **Reporting Engine**: Provides detailed execution traces for audit logs and vulnerability reports.
-   **Self-Improvement Engine**: Learns from execution successes and failures to refine tool usage and error handling strategies.

By meticulously managing the execution lifecycle, the Execution Engine ensures that BugHunterAI can reliably perform complex security research tasks, adapt to dynamic environments, and continuously generate valuable insights.
